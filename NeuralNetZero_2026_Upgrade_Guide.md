### 3.3 The DeepSeek Data Quality Pipeline

DeepSeek V3.2's key data insight: **increase the ratio of mathematical and programming samples, minimize redundancy while maintaining diversity.**

Their document-packing method is relevant at your scale: instead of padding short sequences, pack multiple examples into single sequences separated by special tokens. For your 512-token context window:

```python
def pack_documents(examples, block_size=512, sep_token_id=2):
    """Pack multiple short examples into single sequences."""
    packed = []
    current = []
    
    for example in examples:
        tokens = tokenizer.encode(example['text'])
        if len(current) + len(tokens) + 1 <= block_size:
            current.extend(tokens + [sep_token_id])
        else:
            if current:
                packed.append(current[:block_size])
            current = tokens + [sep_token_id]
    
    if current:
        packed.append(current[:block_size])
    return packed
```

This eliminates padding waste. At your scale with short STEM examples, packing typically gets 3-4 examples per sequence — effectively 3-4x more training signal per gradient step.

## Part 4: Training Objective — The Multi-Token Prediction + Selective Loss Stack

### 4.1 Multi-Token Prediction (DeepSeek V3 / Meta FAIR)

DeepSeek V3 pioneered MTP at production scale and found 85-90% token acceptance rates, confirming the auxiliary heads learn real predictive structure. The key: DeepSeek uses **sequential** MTP (each head predicts conditioned on previous head), not parallel. Their ablations showed sequential outperforms parallel at larger scales.

```python
# In model.py — add to GPT class

class MTHead(nn.Module):
    """Auxiliary head for multi-token prediction."""
    def __init__(self, config):
        super().__init__()
        self.norm = RMSNorm(config.d_model)
        self.proj = nn.Linear(config.d_model, config.vocab_size, bias=False)

class GPT(nn.Module):
    def __init__(self, config):
        super().__init__()
        # ... existing init ...
        
        # Add MTP heads — predict t+2, t+3, t+4
        self.mtp_heads = nn.ModuleList([
            MTHead(config) for _ in range(3)
        ])
        self.mtp_weight = 0.3  # auxiliary loss weight — from DeepSeek ablations

    def forward(self, idx, targets=None):
        # ... existing forward ...
        x = self.transformer(tok_emb)
        x = self.norm_f(x)
        
        logits = self.lm_head(x)
        
        loss = None
        if targets is not None:
            # Main NTP loss
            main_loss = F.cross_entropy(
                logits.view(-1, logits.size(-1)), 
                targets.view(-1)
            )
            
            # MTP auxiliary losses
            mtp_loss = 0
            for i, head in enumerate(self.mtp_heads):
                offset = i + 2  # predict t+2, t+3, t+4
                if targets.size(1) > offset:
                    aux_logits = head.proj(head.norm(x[:, :-offset]))
                    aux_targets = targets[:, offset:]
                    mtp_loss += F.cross_entropy(
                        aux_logits.reshape(-1, aux_logits.size(-1)),
                        aux_targets.reshape(-1)
                    )
            mtp_loss /= len(self.mtp_heads)
            
            loss = main_loss + self.mtp_weight * mtp_loss
        
        return logits, loss
```

**At inference, discard the MTP heads entirely.** They add 0 cost to inference, only to training. Parameter overhead is ~3 × (d_model × vocab_size) = ~1.2M extra params during training only.

### 4.2 Selective Token Loss — The RHO-1 Approach

Not all tokens deserve equal gradient weight. "The" appearing after "Newton's" is nearly deterministic — predicting it correctly teaches nothing. "9.8" appearing after "gravitational acceleration on Earth equals" is informationally rich.

```python
def compute_token_weights(token_ids, tokenizer, domain_vocab_file='stem_vocab.json'):
    """Weight tokens by information content. STEM terms get higher weight."""
    import json
    
    with open(domain_vocab_file) as f:
        stem_terms = set(json.load(f))  # your domain-specific high-value tokens
    
    weights = torch.ones_like(token_ids, dtype=torch.float)
    
    for i, token_id in enumerate(token_ids.flatten()):
        token_str = tokenizer.decode([token_id.item()])
        # Numbers, formulas, domain terms get 2x weight
        if any(c.isdigit() for c in token_str) or token_str.strip() in stem_terms:
            weights.flatten()[i] = 2.0
        # Common filler gets 0.5x weight  
        elif token_str.strip().lower() in {'the', 'a', 'an', 'of', 'is', 'are', 'and', 'or'}:
            weights.flatten()[i] = 0.5
    
    return weights.view(token_ids.shape)

# In training loop, replace standard cross_entropy with:
def weighted_cross_entropy(logits, targets, weights):
    loss = F.cross_entropy(logits.view(-1, logits.size(-1)), targets.view(-1), reduction='none')
    weighted_loss = (loss * weights.view(-1)).mean()
    return weighted_loss
```

Build `stem_vocab.json` from your domain: extract all tokens whose frequency in STEM texts is significantly higher than general English. These are your high-signal tokens.

**Important finding from 2025 research:** Use this weighting for fine-tuning/SFT, but keep full NTP for pretraining. The noise in standard NTP acts as a regularizer during pretraining — removing it during pretraining hurts generalization. Apply selective weighting after the main pretraining run.

### 4.3 Future Summary Prediction (2025 Research — Mahajan et al.)

Add an auxiliary head that predicts a bag-of-words summary of the next 50 tokens. This gives the model a planning signal — it must represent "where this sequence is heading" in its hidden state.

```python
class FSPHead(nn.Module):
    """Future Summary Prediction head."""
    def __init__(self, config, window=50):
        super().__init__()
        self.norm = RMSNorm(config.d_model)
        self.proj = nn.Linear(config.d_model, config.vocab_size, bias=False)
        self.window = window
    
    def compute_bow_target(self, token_ids, position):
        """Multi-hot vector of tokens in next `window` positions."""
        future_tokens = token_ids[position:position + self.window]
        bow = torch.zeros(token_ids.new_zeros(1).shape[0], self.proj.out_features)
        bow[0].scatter_(0, future_tokens.unsqueeze(0), 1.0)
        return bow / (bow.sum() + 1e-8)  # normalize
```

## Part 5: Architecture Upgrades — Staying at 1M Params

### 5.1 Sparse MoE in the MLP (DeepSeek / GLM-5 / Kimi K2 Pattern)

Every top model in 2026 uses MoE. The reason is mathematically clean: you get 4x the parameter capacity for the same inference compute. GLM-5 (744B total, 40B active) achieves this at massive scale. Your version:

```python
class SparseMoEMLP(nn.Module):
    """Replace SwiGLUMLP with sparse MoE. Same inference compute, 4x capacity."""
    def __init__(self, config, n_experts=4, n_active=1):
        super().__init__()
        self.n_experts = n_experts
        self.n_active = n_active
        
        # 4 expert MLPs instead of 1 — each at 1/4 the hidden size
        # Total params: same as original SwiGLUMLP
        expert_hidden = config.ffn_dim // n_experts * n_active  
        self.experts = nn.ModuleList([
            SwiGLUMLP(config, hidden_dim=expert_hidden) 
            for _ in range(n_experts)
        ])
        
        # Router — tiny overhead
        self.router = nn.Linear(config.d_model, n_experts, bias=False)
        
        # One shared expert (DeepSeek innovation) — always active
        self.shared_expert = SwiGLUMLP(config, hidden_dim=expert_hidden // 2)
    
    def forward(self, x):
        B, T, C = x.shape
        
        # Route each token to top-1 expert
        router_logits = self.router(x)  # (B, T, n_experts)
        router_probs = F.softmax(router_logits, dim=-1)
        expert_idx = router_probs.argmax(dim=-1)  # (B, T)
        
        # Dispatch
        out = torch.zeros_like(x)
        for i, expert in enumerate(self.experts):
            mask = (expert_idx == i)
            if mask.any():
                out[mask] = expert(x[mask])
        
        # Add shared expert output (DeepSeek's auxiliary-loss-free trick)
        out = out + self.shared_expert(x)
        
        return out
```

The **shared expert** is DeepSeek's key innovation for MoE stability. It handles common knowledge that all tokens need, while specialized experts handle domain-specific reasoning. This is why DeepSeek V3's auxiliary-loss-free load balancing works — the shared expert prevents routing collapse.

Result: Your model now has the capacity of a ~4M param model at 1M param inference cost.

### 5.2 Multi-Head Latent Attention — MLA (DeepSeek / Kimi K2 / GLM-5)

MLA is the single most replicated architectural innovation of 2025. DeepSeek pioneered it, then Kimi K2, GLM-5, and Ling 2.5 all adopted it. The idea: compress the KV cache via low-rank projection.

At your scale (no KV cache concerns for training), the benefit is different: MLA forces the model to compress query/key information through a bottleneck, which acts as a powerful regularizer and produces richer attention representations per parameter.

```python
class MLAttention(nn.Module):
    """Multi-Head Latent Attention — DeepSeek V3 innovation."""
    def __init__(self, config):
        super().__init__()
        self.n_head = config.n_head
        self.d_head = config.d_model // config.n_head
        
        # Latent bottleneck dimension (1/4 of d_model)
        self.d_latent = config.d_model // 4
        
        # Compress KV to latent space
        self.kv_down = nn.Linear(config.d_model, self.d_latent, bias=False)
        
        # Expand back for K and V separately
        self.k_up = nn.Linear(self.d_latent, config.d_model, bias=False)
        self.v_up = nn.Linear(self.d_latent, config.d_model, bias=False)
        
        # Q projection stays standard
        self.q_proj = nn.Linear(config.d_model, config.d_model, bias=False)
        self.o_proj = nn.Linear(config.d_model, config.d_model, bias=False)
    
    def forward(self, x):
        B, T, C = x.shape
        
        # Compress KV through latent bottleneck
        kv_latent = self.kv_down(x)  # (B, T, d_latent) — the key innovation
        
        k = self.k_up(kv_latent).view(B, T, self.n_head, self.d_head).transpose(1, 2)
        v = self.v_up(kv_latent).view(B, T, self.n_head, self.d_head).transpose(1, 2)
        q = self.q_proj(x).view(B, T, self.n_head, self.d_head).transpose(1, 2)
        
        # Apply RoPE to q and k
        # ... (use existing RoPE implementation) ...
        
        # Flash attention
        out = F.scaled_dot_product_attention(q, k, v, is_causal=True)
        out = out.transpose(1, 2).contiguous().view(B, T, C)
        return self.o_proj(out)
```

Param savings from MLA vs standard attention at your scale: negligible. The benefit is the representational bottleneck forcing more compressed, generalizable attention patterns.

### 6.3 The MuonClip Innovation (Kimi K2)

Kimi K2 introduced MuonClip — Muon optimizer with QK-Clip for training stability. You already use Muon. Add QK-Clip:

```python
# In optim.py, after Muon step, add:
def qk_clip(model, max_norm=10.0):
    """Clip QK logits to prevent attention entropy collapse."""
    for module in model.modules():
        if hasattr(module, 'q_proj') and hasattr(module, 'k_proj'):
            # Clip the magnitude of Q and K weight matrices
            with torch.no_grad():
                q_norm = module.q_proj.weight.norm()
                k_norm = module.k_proj.weight.norm()
                if q_norm > max_norm:
                    module.q_proj.weight.mul_(max_norm / q_norm)
                if k_norm > max_norm:
                    module.k_proj.weight.mul_(max_norm / k_norm)

# Call after optimizer.step() in training loop
```

This prevents a known instability in Muon-trained models where attention logits blow up at higher learning rates, causing loss spikes. Kimi K2 trained 15.5 trillion tokens with zero loss spikes using this technique.

### 6.4 Learning Rate Schedule — DeepSeek's WSD Schedule

DeepSeek V3 used a Warmup-Stable-Decay schedule instead of cosine decay. Research shows this is better for data-efficient training:

```python
def get_wsd_lr(step, warmup_steps, stable_steps, total_steps, max_lr, min_lr=0):
    """Warmup-Stable-Decay schedule from DeepSeek V3."""
    if step < warmup_steps:
        return max_lr * step / warmup_steps
    elif step < warmup_steps + stable_steps:
        return max_lr  # stable phase — full LR for most of training
    else:
        # Cosine decay only in final phase
        decay_steps = total_steps - warmup_steps - stable_steps
        decay_progress = (step - warmup_steps - stable_steps) / decay_steps
        return min_lr + 0.5 * (max_lr - min_lr) * (1 + math.cos(math.pi * decay_progress))

# Suggested splits for 20k steps:
# warmup: 500 steps
# stable: 16500 steps  
# decay: 3000 steps
```

The stable phase at full LR lets the model deeply learn the dataset before decaying. With cosine-only, you're decaying too early and leaving performance on the table.

### 7.2 Pivotal Token DPO (Phi-4 Innovation)

Microsoft's Phi-4 introduced Pivotal Token Search for DPO. The idea: instead of doing preference optimization on the entire response, identify the **pivotal tokens** — the key decision points where the model chooses between correct and incorrect reasoning — and focus DPO signal there.

Simple version:
```python
def find_pivotal_tokens(good_response, bad_response, tokenizer, model):
    """Find tokens where good and bad responses diverge most."""
    good_tokens = tokenizer.encode(good_response)
    bad_tokens = tokenizer.encode(bad_response)
    
    # Find first significant divergence point
    for i, (g, b) in enumerate(zip(good_tokens, bad_tokens)):
        if g != b:
            return i  # pivotal position
    return min(len(good_tokens), len(bad_tokens))

# Weight DPO loss higher at and after pivotal tokens
```

For your scale, even basic DPO (without pivotal token search) is a substantial win. Generate 500 preference pairs where you use a frontier model to score two different responses to the same question and label which is better.

## Part 9: What the Giants Do That Doesn't Scale Down

Honest list — don't waste time on these:

| Technique | Why it doesn't help at 1M params |
|-----------|----------------------------------|
| RLHF with human raters | Cost of reward model > value of base model |
| Multi-agent reasoning chains | Context window too small, model too weak to be a useful critic of itself |
| Retrieval augmentation (RAG) | Retrieval system overhead >> model itself |
| DeepSeek's DualPipe parallelism | Single GPU — irrelevant |
| FP8 training | RTX 5070 supports it but gains are <5% and complexity is high |
| Constitutional AI | Needs a strong enough base model to reason about principles |
